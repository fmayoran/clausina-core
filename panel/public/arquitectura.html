ClaUsina — Arquitectura técnica y funcional

La plataforma de contenido y operaciones multi-marca. Un motor agnóstico de marca que prepara, aprueba y publica contenido con generación asistida por IA, y una cápsula aislada por cada marca.

Documento vivo Actualizado: 2026-06-23 3 marcas activas Monorepo + 4 submodules

1 · Visión y principio

Una plataforma para que una o varias marcas planifiquen, aprueben y publiquen contenido sin fricción. La automatización prepara y publica; no decide.

Principio rector

Nada sale sin aprobación humana. El contenido se prepara (texto + imagen/video, generado o real), se registra como pieza pendiente, pasa por una aprobación humana con preview, y recién entonces se publica. Si se rechaza, entra en un loop de iteración hasta aprobarse.

Motor agnóstico + cápsula por marca

El motor no sabe de ninguna marca en particular: sabe operar una marca cualquiera, ruteando por proyecto_id. Cada marca es una cápsula independiente (contexto + assets + secretos), aislada de las demás.

◆
Dualidad de ClaUsina. ClaUsina es a la vez el contenedor/motor/agencia que abarca todo, y una marca más (simétrica con Cortafuego y Ardora): la agencia se hace su propio marketing en clausina.ar.

  • Editar dentro del submodule → commit + push ahí (dispara su deploy: Cloudflare en las landings).

3 · Mapa de componentes

Cinco capas: control/agentes, generación IA, orquestación, datos y presentación, sobre la infraestructura del VPS.

Control / Agentes

Claude Code como operador. Skills globales: /creativo (contenido), /editor (pantalla DOOH), /it (infra). Headless por cron/cola para la auto-corrección.

Panel

clausina_panel (EasyPanel) en panel.clausina.ar. Board: Cola de Requerimientos · Pendientes · Publicadas, login por marca. Sirve el media store en /media.

Generación IA

Higgsfield (CLI, plan ultra): imágenes y Reels. Tipografía de marca: HTML → JPG con Chromium/Playwright. La inicia Fer (calidad/NSFW en el loop).

Orquestación — n8n

Webhooks cf-* (paths estables) tras el rebrand a "ClaUsina - X". Crea piezas, notifica, decide, publica, sirve el feed. Postgres + Telegram + Graph API.

Datos — PostgreSQL

Motor crm_pgvector, base propia claude, schema contenido (multi-proyecto). Fuente de verdad de todo el ciclo de contenido.

Cola Redis + workers

Corrección de rechazos migrándose de cron a cola Redis + worker (piloto Cortafuego). Desacopla el disparo del procesamiento.

Presentación — landings

Sitios estáticos por marca (Cloudflare). index.html con SEO/JSON-LD; preview.html (aprobación) y feed de novedades vía proxy same-origin a n8n.

Canales de salida

Instagram (Graph API), pantalla DOOH (VNNOX, Paseo Ardora), web (novedades), Telegram (operación/avisos a Fer).

Media store

Dos niveles: store central clausina-media (interno/trabajo, panel.clausina.ar/media) + CDN de marca (público).

4 · El proceso end-to-end

El camino completo de una pieza de contenido, desde que entra el requerimiento hasta que se publica. Cada paso indica qué pasa (funcional) y cómo se resuelve técnicamente.

1

Entra el requerimiento (intake)

Fer dicta un brief por voz/foto+texto en Telegram, o sube un pedido en el panel; el creativo también puede proponer qué publicar y qué material hace falta.

Técnico: el brief queda en contenido.tg_briefs con proyecto_id y canal_destino (instagram|aviso). Handlers en el VPS: brief_local.sh (voz → Whisper.cpp local) y propuestas_local.sh. Ruteo por marca: cada corrida = una sola cápsula marcas/<slug>/.
2

El agente prepara la pieza

El Director Creativo (o el Editor, para avisos de pantalla) redacta el copy con la voz de la marca y arma el material, generándolo si no hay foto/video real.

Técnico: Claude Code headless (sobre la suscripción, sin API) lee contexto/CONTEXTO_MARCA.md de la cápsula. Imagen/Reel con Higgsfield; tipografía de marca renderizada HTML→JPG (Chromium). El material se acondiciona (reencode IG / poster) y entra al circuito.
3

Se registra como pieza pendiente

La pieza queda registrada y a la espera de aprobación, con su versión y su material asociados.

Técnico: webhook cf-crear-pendiente inserta piezas + revisiones (+ media) con estado='pendiente_aprobacion' y un token único. Una corrección entra como revisión nueva bajo la misma pieza. El trigger sync_pieza mantiene estado/revision_vigente.
4

Se le avisa a Fer

Fer recibe la pieza para revisar, con preview y botones de Aprobar / Rechazar / Descartar.

Técnico: cf-pub-notify manda una tarjeta de Telegram (imagen/poster + botones inline) vía bot. Botón "Ver completo" → preview.html?token=… (mockup IG + tarjeta de novedad), cargado por proxy same-origin para funcionar incluso dentro de Gmail. Identidad de los avisos: agencia ClaUsina (celeste).
5

Fer decide

Aprueba, rechaza con un motivo, o descarta. Es el único disparador real de publicación.

Técnico: los botones pegan a cf-pub-decide (valida el token, cambia el estado). aprobada dispara la publicación; rechazada guarda el motivo y abre el loop; descartada es estado terminal.
6

Se publica solo

Al aprobar, la pieza sale al canal correspondiente y queda registrada como publicada.

Técnico: Instagram → cf-pub-publish con Graph API (crear container imagen/Reel/carrusel → poll de status_code hasta FINISHED → media_publish), guarda ig_post_id + ig_permalink → publicada. Aviso DOOH → publicada = "en pantalla" (sin API externa; lo levanta VNNOX). La web lee las publicadas vía cf-novedades.
↻
Loop de corrección. Un rechazo crea una revisión nueva (misma pieza) y vuelve al paso 4. La auto-corrección lo resuelve sola para texto y visual (regenera con Higgsfield + tipografía de marca) y reenvía; escala a Fer al 5º intento. Históricamente por rutina_local.sh (cron cada 5 min); migrándose a cola Redis + worker. Soporte: cf-rechazos-pendientes / cf-marcar-procesado / cf-avisar.

5 · Canales

Un intake único (la Cola de Requerimientos, agnóstica del canal) separa el qué del dónde. La misma máquina de estados y aprobación vale para todos los canales.

Instagram

Feed, carrusel, Reels e Historias vía Graph API. Aprobar → cf-pub-publish. Métricas y Collab en el panel. Token IG en credencial cifrada de n8n (no en git ni en la DB).

Pantalla DOOH (Paseo Ardora)

Cartel de calle vertical (2×3 m, mudo). El /editor arma spots 2:3 ~10s. Aprobar = "en pantalla". Integración VNNOX (botón "A la pantalla", device T1-4G).

Web / Novedades

Landings estáticas en Cloudflare. El feed de novedades hace fetch a cf-novedades (n8n → DB) y renderiza; al publicar una pieza aparece sola, sin redeploy. Home conserva su SEO estático.

Telegram

Canal operativo: tarjetas de aprobación con botones, ingreso de briefs por voz/foto y avisos a Fer. Workflow cf-telegram + bot.

Claves del diseño

  • Publicación = revisión con estado='publicada'. El feed lee las publicadas filtrando por revision_vigente y canal='instagram'.
  • Trazabilidad del loop: cada corrección es una revisión nueva (nro+1) bajo la misma pieza; el texto queda como historial, la media se reemplaza (no acumula peso).
  • Estados (estado_pub): borrador → pendiente_aprobacion → aprobada → publicada · más rechazada y descartada. derivado_en marca lo escalado a Fer.
  • Triggers: sync_pieza mantiene estado y revision_vigente de la pieza. DDL en scripts/db/.

Asador urbano (Ranelagh). Marca piloto, donde se valida todo el circuito. @cortafuego.ar Worker
